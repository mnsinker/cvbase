# ⚐ 为什么产品转 → 研发？

我想对完整的解决方案负责。

不只是定义问题，而是对整个过程负责——从问题定义，到架构设计、实现与交付。

- 理解问题
- 设计系统
- 构建解决方案
- 交付结果

研发让我拥有了闭环的能力。


# ⚐ 我的产品背景能为研发带来什么？

十年的产品经理经历让我积累了三种能力。

**定义正确的问题**

    - 为什么重要：如果问题没有被清晰定义，就无法衡量成功。
    - 缺少它会怎样：评估会变得主观，范围会不断扩大，项目也会难以收尾。

**为变化而设计**

    - 为什么重要：业务需求永远在不断演变。
    - 缺少它会怎样：每一项新需求都会增加交付时间、维护成本和架构复杂度。

**建立共同理解**

    - 为什么重要：不同团队有不同的视角和目标。一个成功的解决方案，需要在开始实现之前建立跨团队的共识。
    - 缺少它会怎样：团队会解决不同的问题，决策会变得不一致，最终的解决方案也很难让不同团队所拥护。


# ⚐ 我构建过什么？

我并不是从一份宏伟的路线图开始的。
我从解决眼前真实的问题开始。
直到后来，才意识到它们其实属于一个更大的系统。

**Decision Engine**
- **问题**：业务决策难以解释、验证和维护。
- **构建**：一个由策略驱动的决策引擎，让每一步推理都清晰明确且可追溯。
- **发现**：可靠的决策依赖结构化的知识。

**Question Forge**
- **问题**：教育内容被锁在 PDF 文档中。
- **构建**：一套将 PDF 转换为结构化 Question Object 的 AI 工作流。
- **发现**：AI 生成的代码很快就会变得难以维护。这促使我开始探索 AI Coding Governance。

**CV Base + Compiler**
- **问题**：每次更新简历，都意味着要重复改写相同的经历。
- **构建**：一个基于叙事的知识系统，让简历和作品集来自同一个源头。
- **发现**：在渲染文档时要保留并体现信息层级结构，需要一个**编译器**。

