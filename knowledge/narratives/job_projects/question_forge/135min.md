# 1-Minute Version

Question Forge is a document-understanding system that converts educational documents such as PDFs and scanned exam papers into structured Question Objects.

The problem it solves is that most schools and tutoring organizations still organize their question banks by documents rather than by individual questions. Teachers can read the content, but they cannot efficiently search, reuse, analyze, or personalize questions because the questions are trapped inside PDFs.

The system changes the unit of storage from a document to a structured Question Object containing fields such as the stem, options, answer, explanation, and diagram assets. The current pipeline uses OCR for localization and a vision-language model for semantic parsing. It detects question boundaries, extracts diagrams, parses each question into a structured schema, and exports JSON and HTML previews.

One of the most valuable engineering lessons came from debugging diagram ownership. A diagram appearing under the wrong question initially looked like a diagram detection bug, but after tracing the pipeline I discovered it was actually caused by incorrect question boundaries. That led me to build audit artifacts and intermediate traces so I could locate failures before modifying algorithms.

The long-term goal is to evolve from document conversion into a structured educational knowledge platform that supports retrieval, AI-assisted question generation, and personalized learning.

# 3-Minute Version

Question Forge is a document-understanding system for educational content.

The motivation came from observing how most educational organizations manage their question banks. They usually organize content by folders and PDF documents. While humans can read those documents, individual questions are difficult to search, compare, tag, or reuse. More importantly, AI systems cannot reason about questions that only exist as images inside PDFs.

The first goal of Question Forge is therefore to convert documents into structured Question Objects. Each Question Object can contain fields such as the question stem, options, answer, explanation, source information, and associated diagram assets. Once questions become structured objects, they can later support search, analytics, recommendation, and AI generation.

Architecturally, the system follows a hybrid OCR and vision-language-model pipeline. OCR is responsible for localization, while the VLM focuses on semantic understanding. The pipeline renders PDF pages into images, detects and recognizes text, identifies question boundaries, creates question-level crops, extracts diagrams, parses the semantic content into a structured schema, and exports JSON together with HTML previews and image assets.

One engineering challenge I spent a lot of time on was diagram ownership. In one version, the diagram from Question 20 also appeared under Question 6. At first I assumed it was a diagram detection problem. After generating ownership reports and pipeline traces, I discovered the real issue was an oversized Question Crop. Multiple questions were being parsed from the same crop, causing downstream metadata to propagate incorrectly. The solution was not changing diagram detection, but treating Question Boundaries as the ownership boundary of the entire pipeline.

Another challenge involved diagram crop quality. Strengthening watermark filtering prevented false positives, but valid diagrams became too tightly cropped. Instead of loosening the detection rules, I separated diagram detection from diagram projection and introduced adaptive padding. This recovered missing labels while keeping the detection logic unchanged.

The current version focuses on document-to-question conversion. The next stage is enriching Question Objects with educational metadata such as knowledge points and difficulty. The long-term vision is to combine Question Objects with Student Context so the system can recommend the right questions for each learner.

# 5-Minute Version

Question Forge is a document-understanding platform that transforms educational documents into structured Question Objects.

The project started with a representation problem rather than an AI problem. Most educational institutions already have thousands of high-quality questions, but they are stored inside PDFs, scanned worksheets, and images. Those documents are easy for teachers to read but difficult for software to search, analyze, or personalize. If the basic unit of storage is a document instead of a question, many downstream capabilities become impossible.

The first objective of Question Forge is therefore to convert document-centric content into question-centric data. A Question Object represents a single question and can include fields such as the stem, options, answer, explanation, diagrams, and, in future versions, educational metadata like knowledge points, prerequisites, and difficulty. Once questions become structured objects, they can support search, retrieval, generation, and personalization.

The current architecture uses a hybrid OCR and vision-language-model pipeline because I found that neither approach alone was sufficient.

OCR provides deterministic localization. It detects text regions, recognizes the text, and generates Question Boundaries. Those boundaries define Question Crops, which become the processing context for everything downstream.

From each Question Crop, the pipeline branches into two paths. One path performs diagram detection and produces diagram assets using OCR masking, connected-component analysis, and filtering. The other path sends the crop to a vision-language model, which extracts semantic information such as the stem, options, answer, and explanation. Finally, the diagram assets and semantic output are merged into an enriched Question Object and exported as JSON and HTML previews.

One of the most important architectural discoveries was that ownership begins with Question Boundaries. During development, the diagram from Question 20 incorrectly appeared under Question 6. Initially I believed diagram assignment was wrong, but after building ownership reports, overlays, and intermediate trace artifacts, I found that the Question 20 crop was too large and also contained Questions 6 and 7. The VLM parsed multiple questions from the same crop, and downstream metadata inherited the wrong ownership. The fix was therefore not diagram detection but correcting Question Crops and making ownership deterministic.

Another engineering story involved diagram quality. After improving watermark filtering, diagrams became cleaner, but labels such as x, y, O, B, and C were clipped. Rather than weakening the detector, I separated the concepts of detection and projection. Detection identifies the core diagram, while projection determines how much surrounding context should be retained. Replacing a fixed crop margin with adaptive padding significantly improved diagram completeness without changing ownership or detection.

Beyond the extraction pipeline, I’ve also started building a reusable Graph Runtime. Instead of coupling relationships directly to educational entities, I model information through a generic pipeline of Info → Association → Node → Relation → Graph. The current implementation uses repository folders as the first domain, but the long-term goal is to reuse the same graph infrastructure for educational concepts, Question Objects, and eventually Student Context.

From a product perspective, the roadmap has three stages. First, convert educational documents into structured Question Objects. Second, enrich those objects with educational metadata and build a searchable question repository. Third, introduce Student Context by recording learning history and compressing it into an AI-friendly representation, allowing the system to recommend questions that match a student’s knowledge state, curriculum, and learning goals.

The biggest lesson from this project is that successful AI systems depend more on stable representations, ownership boundaries, and observable workflows than on model capability alone. OCR and VLM are replaceable components, but the architecture that organizes information into reusable representations is what enables the system to evolve.
