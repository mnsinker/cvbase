# 问题空间

知识缺失

    陈述：
        项目文件存在。
        项目知识不存在。
    
    症状：
        显性知识不存在。
    
        - 架构决策
        - 权衡取舍
        - 发现
        - 能力
    
    根本原因：
        项目信息被记录下来，却从未被提炼为可复用的知识。
    
    结果：
        每一项交付物都必须重新创建。
    
        简历 → 重写
        面试准备 → 重写
        作品集 → 重写
        博客 → 重写


# 能力

## 当前能力

捕获信息

    问题：
        信息是如何被收集的？
    
    流程：
        信息 → 收件箱
    
    示例：
        * GPT 对话
        * 书籍摘录
        * 项目笔记
        ---------------
        → 收件箱
    
    优势：
        - 无摩擦捕获
        - 统一入口


将信息提炼为知识

    问题：
        原始信息如何变成可复用的知识？
    
    流程：
        收件箱 → 叙事
    
    示例：
        Question Forge 项目 → 叙事，其中包含：
        - 问题
        - 约束
        - 权衡取舍
        - 决策
        - 结果
    
    优势：
        - 保留架构推理
        - 减少重复分析
        - 积累可复用知识


生成交付物

    问题：
        知识如何应用于不同目标？
    
    流程：
        叙事 → 简历
        叙事 → 作品集
        叙事 → 面试准备
    
    示例：
        一个叙事
        ↓
        简历 / 作品集 / 面试准备
    
    优势：
        - 单一事实来源
        - 多种交付物
        - 无需重复改写


## 未来能力

提取概念

    问题：
        如何识别可复用的概念？
    
    流程：
        叙事 → 概念
        
    示例：
        叙事
        ↓
        所有权 | 反馈循环 | 策略 | 表征
    
    优势：
        - 改善检索
        - 减少重复概念
        - 建立共享词汇


综合洞察

    问题：
        如何跨多个叙事综合更高层次的洞察？
    
    流程：
        叙事
        + 叙事
        + 叙事
        ----------------
        → 洞察
    
    示例：
        Question Forge
        + 编译器
        + 决策引擎
        ----------------------
        → 洞察：计算之前必须先建立表征
    
    优势：
        - 发现反复出现的模式
        - 连接彼此独立的叙事
        - 生成可复用知识


推荐行动

    问题
        给定当前目标，下一步应该做什么？
    
    流程
        洞察
        + 当前目标
        + 当前上下文
        (+ 之前的结果)
        ---------------------
        → 推荐行动
    
    说明
        通过运行时策略生成实现（见架构）。
        
    示例
        - 面试时应强调哪个叙事
        - 哪个项目值得进一步投入
        - 接下来应该提升哪些技能
    
    优势
        - 减少猜测
        - 优先处理高影响力工作
        - 改善职业决策


捕获结果

    问题：
        发生了什么？
    
    流程：
        行动 → 结果
    
    示例：
        - 记录面试反馈
        - 记录申请结果
           
    优势：
        - 保留现实世界的反馈
        - 识别反复出现的弱点
        - 减少重复犯错


从结果中学习

    问题：
        知识系统如何持续改进？
    
    流程：
        结果 → 通过反思 → 更新后的叙事 → 更新后的洞察
    
    示例：
        面试反馈 → 更新后的叙事 → 更新后的洞察
    
    优势：
        - 积累经验
        - 持续改善知识
        - 提升决策质量

# 架构

知识系统由三个阶段组成：

1. 知识生成
2. 知识消费
3. 知识演化

它们共同形成一个持续学习的循环。

## 总体流程

```json
信息
↓
收件箱
↓
叙事
↓
概念
↓
洞察
↓
运行时策略
↓
行动
↓
结果
↓
反思
↓
更新后的叙事
```

以下章节将解释这一生命周期的每个阶段。


## 知识生成

问题：

    原始信息如何变成可复用的知识？

流程：

    ```json
    信息
       │
       │ 捕获
       ▼
    收件箱
       │
       │ 提炼
       ▼
    叙事
       │
       │ 提取
       ▼
    概念
       │
       │ 生成
       ▼
    洞察
    ```

关键组件：

    信息
        从书籍、对话、项目和日常工作中捕获的原始观察。
    
    收件箱
        知识提炼之前的临时收集层。
    
    叙事
        供下游复用的主要知识表征。
      
    概念
        从单个叙事中提取、用于 AI 检索的语义索引。
    
    洞察
        从多个叙事中综合出的更高层次原则。

关键发现：

    知识不是被直接捕获的，而是通过提炼生成的。
    提炼将信息与以下内容连接起来：
        - 上下文
        - 约束
        - 权衡取舍
        - 示例
        - 过往经验
    没有这些连接：
        - 信息仍然只是信息。
        - 它永远不会变成知识。



## 知识消费
问题：
    知识如何针对具体目标转化为决策？

流程：

```json
洞察 + 当前目标 + 当前上下文
  │
  │ 推理
  ▼
运行时策略
  │
  │ 优先排序
  ▼
推荐行动
```

关键组件：

    洞察
        跨多个叙事积累的、可长期复用的知识。
    
    当前目标
        系统试图实现的目标。
    
        示例：
        - 简历
        - 作品集
        - 面试准备
    
    当前上下文
        与当前情境相关的信息。
        示例：
        - 目标职位
        - 面试历史
        - 可用项目
    
    运行时策略
        针对当前目标和当前上下文生成的临时决策策略。
    
    推荐行动
        根据运行时策略产生的具体下一步行动。
        示例：
            - 强调编译器。
            - 弱化产品管理。

## 知识演化

问题：

    知识系统如何随时间持续改进？

流程：
```json
结果
↓
反思
↓
更新后的叙事
↓
更新后的概念
↓
更新后的洞察
```

关键组件：

    结果
        之前的行动产生的现实结果。
        示例：
            - 面试反馈
            - 申请结果
            - 作品集表现
    
    反思
        分析结果以识别：
            - 哪些做得有效
            - 哪些没有奏效
            - 原因是什么
    
    更新后的叙事
        记录新的观察、决策和推理。
    
    更新后的概念
        更新用于未来检索的语义索引。
    
    更新后的洞察
        改进更高层次的原则。
    
    关键发现
        反馈本身不会改善知识。
        只有通过反思，知识才会演化。
        结果只有在被解释、连接和提炼之后，才会变成知识。


## 通用计算模式

问题：
    为什么
        - CV Base
        - 决策引擎
        - 工作流引擎
        - 图引擎
    共享相同的架构？

模式：
```json
输入
↓
表征
↓
转换
↓
计算
↓
输出
↓
反馈
↓
更新后的表征
```



示例 1：编译器
流程
```json
Markdown（# 问题空间）
↓
AST（H1，“问题空间”）
↓
语义分析（检测：标题层级、父子所有权、语义类型）
```

实例
```json
# 问题空间
↓
H1（问题空间）
↓
检测：标题层级、父子所有权、语义类型
```

为什么需要表征？

    没有 AST，语义分析就无法确定所有权、层级或语义。

示例 2：CV Base
流程
```json
经历
↓
叙事
↓
知识产品
```

实例
```json
CV Base 项目经历
↓
叙事
↓
简历 / 作品集 / 概念 / 洞察
```

为什么需要表征？

    没有叙事，就无法从项目经历生成知识产品。


示例 3：Question Forge
流程
```json
PDF
↓
Question Object
↓
搜索 / 生成 / 个性化
```

实例
```json
试卷.pdf
↓
Question（题干、图示、选项、难度级别）
↓
题目搜索 | AI 题目生成 | 个性化练习
```
为什么需要表征？

    没有 Question Object，题目就无法支持检索、生成和个性化。


示例 4：决策引擎
流程
```json
上下文
↓
状态
↓
策略评估
```

实例
```json
用户：“我想退款”
↓
OrderSummary（status="shipped"，delivered_days=3）
↓
允许退款
```

为什么需要表征？

    没有状态，策略评估就无法处理原始上下文。


关键结论：

    结构化表征是计算的前提。
    不同领域需要不同的表征，但计算原则保持不变。



# 故事
这些故事记录了构建 CV Base 编译器过程中产生的工程发现。
它们关注系统如何被设计、建模决策如何形成，以及实现问题如何被拆解。


## 故事：模块设计——设计编译器模块

背景

    渲染器需要确定性的输入。
    LLM 生成的结构不够确定。
    编译器因此成为原始内容与渲染之间的确定性层。

模块边界

```json
原始内容
      │
      ▼
+----------------+
|      编译器    |
+----------------+
      │
      ▼
结构化表征
      │
      ▼
+----------------+
|      渲染器    |
+----------------+
      │
      ▼
    HTML
```

    为什么需要独立的编译器？
    - 编译器不应该了解任何作品集相关信息。
    - 它只应该将原始内容转换为结构化表征。

设计方法：

    第 1 步：识别输入和输出
        目的是什么？模块应该产出什么？
    
    第 2 步：问题分解
        必须回答哪些问题？
        应该按照什么顺序回答？
        这些问题定义了流水线。
    
    第 3 步：实体设计
        哪些实体可以表示每个阶段？
    
    第 4 步：函数设计
        哪个函数负责将一个实体转换为下一个实体？

编译器流水线

    Markdown                 原始输入
        ↓
    ParsedLine               每一行是什么？
        ↓
    节点树                   谁拥有谁？
        ↓
    带注解的节点树           领域语义是什么？
        ↓
    重写后的节点树           领域语义如何改变所有权？
        ↓
    渲染输出                 应该如何展示？

关键发现

    目标
    ↓
    问题
    ↓
    实体
    ↓
    函数
    
    从终点开始。不要从实现细节开始，而要从目的地开始设计。


接下来

    流水线定义了需要哪些表征。
    下一个问题是：
        - 如何决定什么值得拥有自己的表征？
