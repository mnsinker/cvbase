# 1-minute version

CV Base is a knowledge system I built to solve a very practical problem: every time I updated my resume, prepared for an interview, or built a portfolio, I had to reconstruct the same project experience from scratch.
The root problem was that project files existed, but reusable project knowledge did not. Architectural decisions, trade-offs, discoveries, and outcomes were scattered across notes and conversations.
So I created a narrative-based system. Raw information first enters an inbox, then gets distilled into structured project narratives. From the same narrative, the system can generate different outputs, including resumes, interview preparation, and portfolio content.

I also built a **compiler** that transforms Markdown into a structured node tree before rendering it. This preserves hierarchy, ownership, and semantic information across different outputs.
The key idea behind CV Base is simple: capture knowledge once, structure it properly, and reuse it across many goals.

# 3-minute version

CV Base started from a recurring problem in my own job search.

Every time I needed to update my resume, prepare for an interview, or build a portfolio, I had to reconstruct the same project experience again. Sometimes I forgot important decisions. Sometimes I remembered only the final implementation but not the reasoning, trade-offs, or discoveries behind it.

I realized the problem was not that project information was missing. I already had code, notes, GPT conversations, and documents. The real problem was that this information had never been distilled into reusable knowledge.

So I built CV Base as a knowledge-first system.

The flow begins with raw information, such as project notes, conversations, and reading highlights. That information enters an inbox and is then distilled into a narrative. The narrative captures the problem, constraints, decisions, trade-offs, architecture, capabilities, and outcomes of a project.

That narrative becomes the single source of truth. From it, the system can generate different deliverables, such as a resume, portfolio, and interview preparation materials, without rewriting the same experience each time.

While building this, I encountered another engineering problem. The renderer needed deterministic structure, but raw Markdown and LLM-generated outputs were not reliable enough. So I introduced a compiler between the content and the renderer.

The compiler transforms Markdown through several stages:

Markdown
→ Parsed Lines
→ Node Tree
→ Annotated Node Tree
→ Rewritten Node Tree
→ Rendered Output

The node tree preserves hierarchy and ownership. Annotations add domain semantics, while rewrite rules can change how the final document is organized.

One important discovery was that representation must come before rendering or computation. Without a structured representation, the system cannot reliably understand hierarchy, ownership, or meaning.

Today, CV Base supports information capture, narrative generation, and reusable deliverables. The longer-term direction is to extract concepts across narratives, synthesize higher-level insights, recommend actions based on current goals, and learn from real-world outcomes.

So CV Base is not only a resume generator. It is an early version of a personal knowledge and learning system.

# 5-minute version

CV Base is a knowledge system I built to solve a problem I repeatedly experienced while preparing for job applications and interviews.
Every time I updated my resume, prepared a project story, or built a portfolio page, I found myself rewriting the same experience from scratch. The source materials already existed: code, project files, notes, GPT conversations, and previous resume bullets. But the important knowledge was still missing.
For example, the files could show what had been implemented, but they did not clearly preserve:

* why a decision was made;
* what alternatives were considered;
* what trade-offs existed;
* what changed during implementation;
* what the system was capable of;
* and what I learned from the project.

So the root problem was not missing information. It was missing distillation.

I designed CV Base around a simple flow:

Information
→ Inbox
→ Narrative
→ Knowledge Products

Raw information first enters a unified inbox. It can come from project notes, books, conversations, or daily work.
The next stage is narrative generation. A narrative is the main reusable representation in the system. Instead of storing only isolated facts, it preserves the context around a project: the problem, constraints, architecture, decisions, trade-offs, discoveries, and outcomes.
Once the narrative exists, multiple deliverables can be generated from the same source:

Narrative
→ Resume
→ Portfolio
→ Interview Preparation

This creates one source of truth and removes repeated rewriting.

However, while implementing the portfolio renderer, I found that content reuse alone was not enough. The renderer needed deterministic structure. Raw Markdown could express headings and indentation, but it did not explicitly represent ownership, hierarchy, or domain meaning. LLM-generated structures were also too inconsistent to serve directly as renderer input.

That led me to build a compiler.

The compiler creates a deterministic layer between raw content and presentation:

Raw Content
→ Compiler
→ Structured Representation
→ Renderer
→ HTML

Its internal pipeline is:

Markdown
→ ParsedLine
→ Node Tree
→ Annotated Node Tree
→ Rewritten Node Tree
→ Rendered Output

Each stage answers a different question.

ParsedLine answers: what is on this line?

The node tree answers: who owns whom?

Annotations answer: what does this node mean in the domain?

Rewrite rules answer: how should those semantics affect the final structure?

The renderer then answers: how should the structure be presented?

One of the most important engineering discoveries from this project was how to design entities.

Initially, I thought an entity deserved its own representation if it had a lifecycle, an ID, or a stable structure. Later, I realized those are consequences of introducing an entity, not the criteria.

The better criterion is connectivity.

If a concept is reused by many parts of the system, or changing it would affect many downstream consumers, it deserves its own representation.

For example, I originally designed the compiler around portfolio-specific concepts such as sections and subsections. That worked for one renderer but could not support other content types. I then redesigned the abstraction around a generic Node:

Node
├── ast_type
├── semantic_type
├── content
└── children

This allowed the compiler to remain stable while different renderers could consume the same structure.

I also learned to design functions by starting from the desired output rather than immediately writing control flow.

For example, when converting parsed lines into a tree, my first implementation accumulated if statements, loops, stack updates, and special cases. I stepped back and asked one simpler question:

Before a node can appear in the final tree, what must happen?

The answer was that it must first find its parent.

That clarified the pipeline:

Find Parent
→ Update Tree
→ Update Stack

This made the implementation easier to reason about and extend.

Today, CV Base supports three main capabilities:

First, capturing information through a unified inbox.

Second, distilling information into structured project narratives.

Third, generating reusable outputs such as resumes, portfolios, and interview materials.

The future direction is to extend the system from knowledge storage into a learning loop:

Narrative
→ Concept
→ Insight
→ Runtime Policy
→ Action
→ Outcome
→ Reflection
→ Updated Narrative

Concepts would help retrieve related knowledge across projects. Insights would synthesize patterns across multiple narratives. Runtime policies would combine long-term knowledge with the current goal and context to recommend actions. Outcomes and reflection would then update the knowledge system.

So the broader goal of CV Base is not simply to generate a better resume.

It is to build a system that captures experience, turns it into reusable knowledge, applies that knowledge to real decisions, and improves through feedback.
