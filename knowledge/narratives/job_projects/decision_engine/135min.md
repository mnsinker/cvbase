# 1-Minute Version

AI Decision Engine is a graph-based system I built to make business decisions explicit, testable, and reusable.

The initial use case was refund evaluation. A user can ask whether an order is refundable, and the system retrieves the required information, evaluates the refund policy, and returns a decision with an explanation.

My first implementation repaired missing dependencies during execution. Although it worked, planning and execution were tightly coupled, making the runtime difficult to understand and validate.

I then separated the system into four independent layers: a planner, a validator, a tool execution layer, and a policy layer. The planner builds an execution plan before runtime, the validator checks graph correctness, tools produce facts, and policies transform those facts into business decisions.

Another important improvement was making tool signatures the single source of truth. Instead of maintaining dependency information in multiple places, the system derives graph metadata automatically.

The biggest lesson from this project is that architecture is the management of complexity. Every major refactor moved uncertainty out of runtime execution and into explicit, testable system boundaries.

# 3-Minute Version

AI Decision Engine is a graph-based decision system I built to explore how business decisions can be represented explicitly instead of being hidden inside application code.

The first use case was refund evaluation.

A user asks, “Can order 123 be refunded?” The system needs to understand the request, retrieve the order information, evaluate the refund policy, and return a decision together with the reasoning.

My initial implementation used a reactive runtime loop. Before executing each tool, the runtime checked whether all required inputs already existed. If a dependency was missing, it located the tool that could produce it, inserted that tool into the execution sequence, and continued.

Although this worked, it exposed an architectural issue. Planning and execution happened simultaneously, so the execution path could change while the workflow was already running. That made the system harder to understand, test, and debug.

The first architectural improvement was introducing a planner.

Each tool declares the entities it requires and the entity it provides. Those relationships form a dependency graph. Given a target entity such as RefundDecision, the planner traverses the graph and generates the complete execution plan before runtime begins.

The next issue was graph correctness. The original runtime included cycle detection, but validation still happened during execution and only covered the active path. I introduced a dedicated validator that checks cycles, missing producers, and dependency integrity before execution starts.

I also separated facts from decisions. Tools retrieve business facts, such as OrderSummaryDTO, while policies consume those facts and return decision objects like RefundDecisionDTO. This keeps business rules independent from service implementations.

Finally, I removed duplicated metadata. Originally, dependency information existed in the function signature, the tool registry, and the entity mapping. I made the function signature the single source of truth and derived the graph metadata automatically.

Today the system consists of semantic resolution, graph planning, validation, tool execution, policy evaluation, and response generation.

The main engineering principle behind the project is that architecture should progressively reduce runtime uncertainty by introducing clear and independently testable boundaries.

# 5-Minute Version

AI Decision Engine is a graph-based decision system I built to study how business decisions can become explicit computational objects rather than remaining hidden inside application code or prompts.

The project began with a simple refund use case.

A customer asks, “Can order 123 be refunded?” To answer that question, the system must determine the user’s intent, retrieve the relevant business information, evaluate company policy, and explain the final decision.

The first implementation relied on a reactive execution loop.

The runtime started from the requested task and executed tools one by one. Before executing a tool, it checked whether all required inputs were already available. If a dependency was missing, it identified the tool capable of producing that dependency, inserted it into the active execution list, and continued execution.

Conceptually, the flow looked like this:

Current Step
→ Find Missing Dependencies
→ Insert Missing Steps
→ Resolve Parameters
→ Execute Tool

This design was flexible because it could dynamically repair incomplete execution plans. However, it also created a significant architectural problem.

The runtime was responsible for both deciding what should happen and executing those decisions. Because missing steps could be inserted dynamically, the execution graph was not completely known before runtime started. That made reasoning, visualization, testing, and debugging much more difficult.

The first major architectural evolution was introducing a dedicated planner.

Each tool explicitly declares what it requires and what it provides. Those contracts naturally form a dependency graph.

Given a target entity such as RefundDecision, the planner traverses the graph backward, identifies every required dependency, and produces a complete execution plan before runtime execution begins.

The runtime therefore becomes much simpler:

Target Entity
→ Planner
→ Execution Plan
→ Runtime Execution

The important realization was that planning and execution answer different questions.

Planning answers:

What needs to happen?

Execution answers:

Execute the approved sequence.

Once the planner existed, another issue became obvious.

Having a graph does not guarantee that the graph is correct.

The original runtime already detected cycles, but only while executing. It could not verify the entire dependency graph beforehand or detect missing producers.

To solve this, I introduced a dedicated validator.

The validator performs two categories of checks:

* cycle validation to ensure dependency traversal cannot loop;
* integrity validation to ensure every required entity has a producer and every dependency can be satisfied.

This created another explicit architectural boundary:

Build Graph
→ Validate Graph
→ Plan
→ Execute

The third architectural improvement involved metadata management.

Initially, dependency information was duplicated across multiple locations:

* the function signature;
* the tool registry;
* the entity-to-tool mapping.

Maintaining three sources of truth increased maintenance cost and created opportunities for inconsistency.

I changed the function signature into the single source of truth. A schema derivation step automatically extracts the required and provided entities and builds the dependency graph directly from type annotations.

This eliminated duplicated metadata and significantly simplified graph construction.

The fourth architectural decision was separating tools from policies.

Tools answer factual questions:

* What is the order status?
* What is the customer profile?
* What information exists?

Policies answer business questions:

* Given these facts, what decision should be made?

For example, the refund policy consumes an OrderSummaryDTO and returns a RefundDecisionDTO.

Keeping policy evaluation separate allows business rules to evolve independently from backend services while remaining easy to test.

Today, the complete runtime consists of:

User Request
→ Semantic Resolution
→ Target Entity
→ Planner
→ Validator
→ Tool Execution
→ Policy Evaluation
→ Decision DTO
→ Response Generation

The implemented use case is refund evaluation, but the same architecture naturally extends to approval workflows, marketing decisions, and other policy-driven business systems.

The most important engineering lesson from this project is that architecture is fundamentally about managing complexity.

Every major refactor followed the same direction:

* move planning out of execution;
* move validation out of execution;
* move metadata out of manual wiring;
* move business policies out of tools.

Each change reduced runtime uncertainty and created clearer, independently testable system boundaries.
