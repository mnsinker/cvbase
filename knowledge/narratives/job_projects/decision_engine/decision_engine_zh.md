# 问题空间

## 当前问题

决策逻辑是碎片化的

    组织每天都在做决策。然而，这些决策背后的逻辑通常分散在：
        - Excel 表格
        - 演示文稿
        - 业务规则
        - 代码
        - 部落知识
    因此，决策变得难以理解、治理、测试和改进。

AI 决策引擎：将决策逻辑从隐式流程转变为显式系统。



## 未来问题

执行决策还不够

    做出决策只是第一步。组织还必须确定：
        上下文 + 策略 + 目标 → 下一步最佳行动
    并持续从结果中学习。

长期目标：构建一个能够随着时间推演、执行、评估和改进决策的决策系统。

# 能力

## 当前能力

决策上下文
    定义决策所需的信息。

    支持：订单摘要 → 优惠券资格 → 优惠券决策
    让决策输入变得明确。

决策规划
    从依赖关系中推导执行顺序。

    支持：目标实体 → 依赖图 → 执行计划
    包括：
        规划
        验证
        依赖管理

决策可追溯性
    让决策行为可观察、可解释。

    支持：执行计划 → 执行轨迹 → 审查
    包括：
        依赖图
        执行轨迹
        运行时输出
    支持决策系统的调试与治理。


## 未来能力

策略治理
    支持策略所有权、生命周期管理、审批和版本控制。

    支持：策略 → 版本 → 审批 → 部署


决策学习
    将决策结果转化为可复用的知识。

    支持：决策 → 结果 → 反馈 → 策略更新
    潜在的更新目标包括：
        - 记忆更新
        - 上下文更新
        - 策略更新
        - 目标复盘
    潜在机制：
        - 评估
        - 实验
        - 优化

决策协调
    协调跨多个上下文、策略和领域的决策。

    支持：上下文 + 策略 + 目标 → 协调行动
    潜在实现：
        单智能体协调
        多智能体协调


# 架构

## 架构目标

构建一个能够从：

    上下文 → 决策

演进至：

    上下文 + 记忆 + 策略 + 目标 → 下一步最佳行动

该架构旨在让决策过程可观察、可测试并可演进。


## 动态流程

    请求
    → 目标实体
    → 规划器
    → 执行计划
    → 运行时
    → 实体生成
    → 策略评估
    → 决策

## 静态结构

上下文层

    职责：定义决策所需的信息
    核心组件：实体

规划层

    职责：从依赖关系中推导执行顺序
    核心组件：规划器、验证器

执行层

    职责：执行决策工作流并生成实体
    核心组件：运行时、工具

策略层
    
    职责：评估业务规则并生成决策
    核心组件：策略

关系层
    
    职责：为实体之间的依赖关系建模
    核心组件：依赖图

可追溯性层

    职责：让决策行为可观察、可解释
    核心组件：执行计划、执行轨迹、运行时输出

# 故事

## 核心发现

先决策，再执行

    架构通过逐步将决策移出运行时而演进。
    示例：
        * 依赖发现——执行前规划
        * 依赖正确性——执行前验证
        * 依赖定义——推导而非重复
        * 层级所有权——将策略与执行分离


## 故事：依赖发现——执行前规划

职业主线：工作流、系统
高级主题：自动化、AI 系统

问题
    
    之前：运行时之前不存在执行路径。
            while 循环：
               - 发现依赖
               - 插入步骤

    结果：流水线黑箱
        * 执行路径无法在执行前被审查
        * 依赖问题只能在执行过程中被发现

发现

    规划和执行是不同的关注点。
        - 规划回答：应该发生什么？
        - 执行回答：让它发生。

结果

    之后：依赖图
         → 规划器
         → 执行计划
    运行时消费执行计划，而不是自行构建执行计划。
    
    优势：
        - 在运行时之前发现依赖问题
        - 在运行时之前审查执行路径

## 故事：依赖正确性——执行前验证

职业主线：系统
高级主题：治理

问题

    之前：
        依赖图已经存在，但正确性仍然在执行过程中检查。
        示例：
            - 环检测
            - 缺少生产者
    
    结果：
        - 图的问题只有在执行开始后才能被发现

发现

    验证不是执行。
        - 规划回答：应该运行什么？
        - 验证回答：它能安全运行吗？
    
    执行应该：假设正确性已经成立。
    验证应该：建立正确性。

结果

    之后：
        验证器
    
    优势：
        - 在执行前发现图的问题

## 故事：依赖定义——推导而非重复

职业主线：信息建模、系统
高级主题：治理

问题

    之前：
        依赖定义存在于多个地方。
        示例：
            工具签名
            工具注册表
            实体映射

    结果：
        - 相同的信息需要多次更新
        - 依赖定义可能逐渐失去同步

发现

    工具签名已经描述了：
        - 输入
        - 输出

    只表示一次信息。其他一切都从中推导。

结果

    之后：
        工具签名 → 推导出的依赖

    优势：
        - 依赖定义只存在于一个地方
        - 图关系不再需要手动同步

## 故事：层级所有权——将策略与执行分离

职业主线：系统
高级主题：治理、AI 系统

问题

    之前：业务规则存在于工具内部。
    结果：策略变更需要修改工具。

发现

    事实回答：什么是真的？
    策略回答：应该发生什么？

    事实和决策的演进速度不同，因此需要不同的所有权边界。

结果

    之后：策略层
    优势：策略变更不再需要修改工具。
